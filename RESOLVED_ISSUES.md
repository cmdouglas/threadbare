# Resolved issues

A log of real bugs found (mostly against a live deployment or a real Discord
server) and fixed, moved out of [`DESIGN.md`](./DESIGN.md) §10 to keep that
section focused on genuinely open risks and open questions. Kept here rather
than deleted, per this project's own convention of documenting what broke and
why the test suite didn't already catch it — useful precedent for future
bugs in the same area, even once the specific issue is closed. Entries are in
the order they were found.

- **`sync-worker`'s config never actually refreshed on `docker compose restart sync-worker`, leaving every wizard-configured channel invisible.** Unlike `web`, which bind-mounts `./.env:/app/.env` read-write so the wizard's writes are visible on the host, `sync-worker`'s compose service had only `env_file: - .env` — Docker materializes that into the container's environment once, at container creation, and `restart` doesn't re-resolve it or recreate the container. `sync_worker/cli.py`'s `load_settings()` call goes through `config.py`'s `reload_env_file()`, but with no `.env` file on disk inside that container at all, the call was a silent no-op. Result: the sync worker kept running on stale blank Discord config, never connected to the gateway, and `discover_channels()` (the only code path that ever sets `channels.is_public`) never ran — so channels stayed at their schema-default `is_public = false` regardless of what the wizard's `/channels` step had correctly set for `indexed`, and the board list rendered empty even though setup had "succeeded." Invisible to the test suite for the same reason the earlier `web`-side version of this bug was (see `write_env_updates`/`reload_env_file`'s own tests): every `sync_worker` config test passes an explicit `env` dict, never exercises the `env is None` → real-file path against an actual bind-mount topology. Fixed by giving `sync-worker` the same bind mount, read-only (it never writes the file, only reads it) — verified manually against a real, isolated Docker Compose stack: with the mount in place, editing `.env` on the host and running a plain `docker compose restart sync-worker` was confirmed to actually pick up the new values, matching what the docs already (now correctly) tell operators to do.

- **The web app's attachment-refresh call had an unverified contract and unconfirmed bot-token support — resolved, confirmed live.** Live-tested against the real test Discord server: posted an image to `#general`, backfilled it via `threadbare-sync-worker`, forced its local `url_expires_at` into the past, then hit `/att/{id}` against a real running `threadbare-web`. `refresh_attachment_urls` successfully called `POST /attachments/refresh-urls` with the bot token (no 401 — bot-token support confirmed) and the response shape matched the implemented assumption exactly (`{"refreshed_urls": [{"original", "refreshed"}, ...]}`); `parse_expiry_from_signed_url` correctly parsed the real returned `ex=` value, which matched the attachment's genuine Discord-side expiry. One real, unrelated bug surfaced by this same exercise — see the next entry.

- **(found during the live test above) `web/db.py`'s `PerRequestConnectionSource` silently discarded every write.** `conn.close()` on a connection with an open transaction does not commit it — the adapter had no explicit `commit()` and wasn't wrapped in psycopg's `async with conn:` idiom, so the attachment-refresh write (and any other web-app write) vanished the moment the connection closed, even though the route returned a normal 302 with no error. Invisible to the test suite: `tests/integration/web/`'s `FakePool` shares one already-open, never-closed connection across an entire test, so a route's writes were visible to later assertions via read-your-own-writes within the same uncommitted transaction — no test ever needed a real commit to "look" correct. Only surfaced by testing against a real, separate verifying connection (a live server, then a dedicated regression test using a second `psycopg.connect()` to check). Fixed by wrapping the yield in `async with conn:`, matching `AsyncConnectionPool.connection()`'s own documented commit-on-success/rollback-on-exception behavior; `tests/integration/web/test_db.py` now guards this specifically, using its own connection (not `FakePool`) to verify persistence. **Lesson for future work in this codebase: `FakePool`-based tests validate route logic and response shape, not whether writes actually persist — any new write path in `web/` needs at least one test structured like `test_db.py`'s, not just route-level assertions.**

- **A real image attachment rendered as a plain filename link instead of inline.** `rendering/attachments.py`'s image check relied solely on `attachments.content_type`, but Discord omits that field for some attachments (older uploads, or server-side detection failures — confirmed by reading discord.py's own source: `content_type: Optional[str] = data.get('content_type')`, a `NotRequired` key in Discord's own payload shape). No existing test covered a missing `content_type` paired with an image-extension filename. Fixed with a fallback to `mimetypes.guess_type(filename)` when `content_type` is absent, since `filename` is the one attachment field Discord always sends.

- **Animated custom emoji and animated-GIF link unfurls both rendered as a static first frame.** Two independent gaps with the same symptom. (1) `rendering/markdown.py`'s animated-custom-emoji handling stripped the `a:` prefix to work around the parser library's missing animated branch, but then hardcoded `animated=False` when rendering — losing the animated bit entirely instead of just working around the parser. Fixed by scanning the raw content for animated-emoji ids separately from the parse step. (2) `rendering/embeds.py` never captured `embed.video` at all — for a "gifv"-type embed (e.g. Tenor/Giphy), `embed.image`/`embed.thumbnail` are only ever a static preview frame; the real animated content is a wholly separate field this project's `EmbedLike` Protocol didn't model. Fixed by capturing `video.url` through the full pipeline (new `embeds.video_url` column, migration `0007_embed_video.sql`) and rendering a `<video autoplay loop muted playsinline>` tag in place of the static image when present. A follow-up gap in the same fix: a real gifv embed commonly carries its static preview in `thumbnail_url`, not `image_url` — the video-vs-image precedence only guarded against `image_url`, so the `thumbnail_url` block (a separate, unconditional check) still rendered alongside the video. Fixed by gating that block on the same "has video" check.

- **(found via a side-by-side screenshot against the real Discord client) A bot/webhook message showed a literal `:game_die:` shortcode instead of the real 🎲 emoji.** `rendering/markdown.py`'s `NodeType.EMOJI_UNICODE_ENCODED` handling had always fallen back to literal shortcode text on the stated assumption that "no local emoji-name lookup exists" — true until the `emoji` package (added for hover tooltips on standard emoji) was added for an unrelated reason and turned out to make resolving a shortcode back to unicode trivial (`emoji.emojize`). Only bots/webhooks hit this path at all — Discord's own client always resolves `:name:` shortcodes to real unicode before sending, but a bot can (and, per the reporting screenshot, does) send the literal text; Discord's client still renders that literal text as the real emoji, so this project now does too via `resolve_shortcode_to_unicode`, falling back to literal text only for a genuinely unrecognized name.

- **(found via a production stack trace on `/board/<id>/continuous/page/12`) A message containing an unmatched `**`, `__`, or ` ``` ` (e.g. a stray "**" in prose, or a pasted code fence someone forgot to close) crashed the whole page with a 500.** Root cause was entirely inside the pinned `discord-markdown-ast-parser==1.0.6` dependency, not this project's code: its `search_for_closer` returns a bare `None` when no closing delimiter is found, but its own docstring promises a `None, None` 2-tuple and every call site (`try_parse_node_with_children`, twice) unconditionally unpacks the result as a tuple — so any unmatched delimiter raised `TypeError: cannot unpack non-iterable NoneType object` instead of falling back to literal text the way Discord's real client does. Invisible to the existing test suite because every markdown test up to this point used delimiters in matched pairs; nothing exercised the unmatched case despite it being a completely ordinary way to type a message. Fixed in `rendering/markdown.py` by patching `discord_markdown_ast_parser.parser.search_for_closer` at import time to coerce `None` into `(None, None)`, restoring the behavior its own docstring already documented, rather than forking the package for a one-line bug in an otherwise-unmaintained (last released 2021) dependency.

- **(found by a mod running the setup wizard fresh) After entering the OAuth client secret for the first time, the "Test login" link the wizard's own success message told them to click wasn't on the page.** `wizard.py`'s `oauth()` view fetches `wizard_state` once at the top of the request, then — only in the POST-success branch — writes `discord_oauth_redirect_uri` into the DB via `update_wizard_state()`, but went on to render the response using that same pre-fetch `state` object, never re-reading it. The template gates the "Test login" link on `state.discord_oauth_redirect_uri`, which is `NULL` until this exact call sets it for the first time, so the very response announcing "click Test login below" was the one response where `state` didn't yet reflect the write it had just made — any later page reload re-fetched fresh state and rendered correctly, making it look like a one-off glitch rather than a reliable bug on every fresh wizard run. No existing test caught it because the only assertion checking for the link's presence (`test_oauth_get_hides_secret_form_after_secret_saved`) did a GET *after* the POST rather than asserting on the POST response itself. Fixed by patching the just-written value into the local `state` dict before rendering, instead of relying on a stale pre-update fetch or an extra round-trip re-query.
